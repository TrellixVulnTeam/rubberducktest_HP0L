1    Generating the image test suite
2    -------------------------------
3    
4    On Linux/unix systems, you can use the `imagemagick-testsuite.sh` script
5    to create an image test suite using the `convert` command line utility. You
6    must have ImageMagick installed ("apt install imagemagick" on debian
7    derivatives). There is also a rule in the Makefile, `make image-testsuite`.
8    
9    A more comprehensive test suite can be generated using Gimp (tested on
10   version 2.8.18). To install the plugin, copy `gimp28-testsuite.py` to your
11   gimp plugins directory, on linux/unix systems usually `~/.gimp-2.8/plug-ins`.
12   You can find the plugin location via the Gimp menu `Edit` - `Preferences` -
13   `Folders` - `Plug-Ins`. Once installed, the plugin should appear in "Tools"
14   menu, named "Kivy image testsuite".
15   
16   Test images must be saved in the `kivy/tests/image-testsuite` directory,
17   after this you can run the test. It is (currently) preferable to run it
18   directly as a script, instead of via `make test`, since the latter won't
19   give you useful debug output.
20   
21       cd kivy/tests
22       python test_imageloader.py | less
23   
24   or to get only the summary report:
25   
26       python test_imageloader.py | grep REPORT | less
27

29   Kivy ImageLoader testsuite
30   --------------------------
31   
32   These tools generate a wide range of images for testing Kivy's ImageLoaders.
33   The pixel data is encoded in the file name, and reproduced "from the sideline"
34   in order to verify the pixel data loaded from file. This is used to expose
35   issues in the Kivy core and underlying provider libraries.
36   
37   The filenames consist of sections separated by an underscore `_`:
38   
39       v0_ <W> x <H> _ <pat> _ <alpha> _ <fmt> _ <testname> _ <encoder> . <ext>
40   
41   Variables are enclosed in pointy brackets. The leading `v0_` indicates that
42   it conforms to version 0 of the test protocol (described in this document)
43   and must be present.
44   
45       v0_5x1_rgb_FF_PNG24_OPAQUE_magick.png
46   
47   This is a 5x1 image, pattern is "rgb", alpha is "FF". `PNG24` is the internal
48   file format name (ImageMagick-specific), and `OPAQUE` is the test we  are
49   performing (drawing opaque pixels only), see test names below.
50   
51   * <pattern> indicates the pixel colors (in order), as they are drawn in the
52     image file.
53   
54   * <alpha> is the global alpha from 00-FF (2 bytes, ascii) which is applied to
55     all pixels except 't' (transparent) which have fixed alpha at 00
56   
57   * <fmt> (aka fmtinfo) is an encoder-specific string with information about the
58     format or process that generated the file. For example, if the same image
59     is saved with and without interlacing, it will contain "I1" and "I0" to
60     distinguish the files.
61     * ImageMagick test suite generator uses magick format name, such as `PNG24`
62     * The Gimp generator plugin adds information about the layer that was
63       exported to form the image:
64         * BPP1G = 1 byte per pixel, gray
65         * BPP2GA = 2 bytes per pixel, gray + alpha
66         * BPP3 = 3 bytes per pixel, rgb
67         * BPP4 = 4 bytes per pixel, rgba
68         * IX = indexed, IXA = indexed+alpha
69         * Note: Indexed images are drawn in RGB or GRAY images (with alpha if
70           needed), and converted to indexed before export. These values
71           represent the layer type at time of export, it affects the parameters
72           used for encoding the output data.
73   
74   * <testname> is a special string that indicates what type of data is expected
75     in the file. Options are OPAQUE, BINARY, ALPHA and repeated for grayscale,
76     GRAY-OPAQUE, GRAY-BINARY, GRAY-ALPHA. We expect BINARY and ALPHA tests to
77     result in an alpha channel (details below)
78   
79   * <encoder> identifies the software that created the file, "magick", "gimp" ..
80   
81   * <ext> is the extension, must be lowercase and match the extensions
82     supported by Kivy image loaders (or they will be ignored)
83   
84   
85   Test names
86   ----------
87   
88   * `OPAQUE` tests opaque pixels (normal RGB) (lossess formats)
89   * `BINARY` tests opaque + fully transparent pixels (GIF etc)
90   * `ALPHA` tests semi-transparent pixels (normal RGBA) (PNG32 etc)
91   * `GRAY-OPAQUE` tests opaque grayscale only (various PNG, tga, )
92   * `GRAY-BINARY` tests opaque grayscale + fully transparent pixels (PNGs)
93   * `GRAY-ALPHA` tests semi-transparent grayscale pixels (TGA, XCF)
94   
95   Patterns must conform to the specific test. For example, the pattern "rgb" has
96   undefined behavior for a grayscale test, since r/g/b can't be represented
97   in grayscale. So all grayscale formats must use 0-9A-F only, and optionally 
98   't' for transparent pixels in GRAY-BINARY/GRAY-ALPHA.
99   
100  
101  | Test name    | Valid pattern characters                                    |
102  | -----------: | :---------------------------------------------------------- |
103  | OPAQUE       | `wxrgbcyp`, and full grayscale\*\*                          |
104  | GRAY-OPAQUE  | `0123456789ABCDEF` (full grayscale)                         |
105  | BINARY       | `t` REQUIRED + `wrgbcyp`, limited grayscale (no 0/x!!!)     |
106  | GRAY-BINARY  | `t` REQUIRED + limited grayscale (no 0/x!!!)                |
107  | ALPHA        | `t`, `wxrbgcyp` and full grayscale\*\*                      |
108  | GRAY-ALPHA   | `t`, `0123456789ABCDEF` (full grayscale)                    |
109  
110  * `**`: While grayscale is supported here, it is generally better to use
111    colors, since all the bytes in a grayscale pixel represented as rgb are
112    identical. For example, 'A' or \xAA\xAA\xAA will pass despite a byte
113    order problem.
114  
115  * `!!!`: In some cases, black color is used for binary transparency. So
116    if you use "0" (or "x"), test_imageloader will expect #000000FF in RGBA,
117    but the pixel becomes transparent (a=00) and test fails. All BINARY tests
118    **MUST** include at least one "t" pixel to ensure that the image is
119    saved with an alpha channel.
120  
121  
122  Pixel values
123  ------------
124  
125  Each character in the pattern represents the color of a single pixel. It is
126  important that you include only the correct type of pixel values for the
127  different test types (see table above). Individual pixel values get their
128  alpha from global setting (<alpha> in filename), but 't' pixels always have
129  alpha = 0x00 regardless of the test's alpha setting.
130  
131  
132      OPAQUE, BINARY, ALPHA (lowercase)
133      -----------------------------------------
134      "w" White (#fff)     "x" Black  (#000)!!!
135      "r" Red   (#f00)     "g" Green  (#0f0)
136      "b" Blue  (#00f)     "y" Yellow (#ff0)
137      "c" Cyan  (#0ff)     "p" Purple (#f0f)
138  
139  
140      GRAY-OPAQUE, GRAY-BINARY, GRAY-ALPHA (uppercase)
141      ------------------------------------------------
142      "0" #000!!!  "1" #111     "2" #222     "3" #333
143      "4" #444     "5" #555     "6" #666     "7" #777
144      "8" #888     "9" #999     "A" #AAA     "B" #BBB
145      "C" #CCC     "D" #DDD     "E" #EEE     "F" #FFF
146  
147      !!! See warnings above regarding BINARY tests
