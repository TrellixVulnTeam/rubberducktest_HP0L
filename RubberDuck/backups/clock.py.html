1    ''' 
2    Clock object 
3    ============ 
4     
5    The :class:`Clock` object allows you to schedule a function call in the 
6    future; once or repeatedly at specified intervals. You can get the time 
7    elapsed between the scheduling and the calling of the callback via the 
8    `dt` argument:: 
9     
10       # dt means delta-time 
11       def my_callback(dt): 
12           pass 
13    
14       # call my_callback every 0.5 seconds 
15       Clock.schedule_interval(my_callback, 0.5) 
16    
17       # call my_callback in 5 seconds 
18       Clock.schedule_once(my_callback, 5) 
19    
20       # call my_callback as soon as possible (usually next frame.) 
21       Clock.schedule_once(my_callback) 
22    
23   .. note:: 
24    
25       If the callback returns False, the schedule will be canceled and won't 
26       repeat. 
27    
28   If you want to schedule a function to call with default arguments, you can use 
29   the `functools.partial 
30   <http://docs.python.org/library/functools.html#functools.partial>`_ python 
31   module:: 
32    
33       from functools import partial 
34    
35       def my_callback(value, key, *largs): 
36           pass 
37    
38       Clock.schedule_interval(partial(my_callback, 'my value', 'my key'), 0.5) 
39    
40   Conversely, if you want to schedule a function that doesn't accept the dt 
41   argument, you can use a `lambda 
42   <http://docs.python.org/2/reference/expressions.html#lambda>`_ expression 
43   to write a short function that does accept dt. For Example:: 
44    
45       def no_args_func(): 
46           print("I accept no arguments, so don't schedule me in the clock") 
47    
48       Clock.schedule_once(lambda dt: no_args_func(), 0.5) 
49    
50   .. note:: 
51    
52       You cannot unschedule an anonymous function unless you keep a 
53       reference to it. It's better to add \\*args to your function 
54       definition so that it can be called with an arbitrary number of 
55       parameters. 
56    
57   .. important:: 
58    
59       The class method callback is weak-referenced: you are responsible for 
60       keeping a reference to your original object/callback. If you don't keep a 
61       reference, the ClockBase will never execute your callback. For 
62       example:: 
63    
64           class Foo(object): 
65               def start(self): 
66                   Clock.schedule_interval(self.callback, 0.5) 
67    
68               def callback(self, dt): 
69                   print('In callback') 
70    
71           # A Foo object is created and the method start is called. 
72           # Because no reference is kept to the instance returned from Foo(), 
73           # the object will be collected by the Python Garbage Collector and 
74           # your callback will be never called. 
75           Foo().start() 
76    
77           # So you should do the following and keep a reference to the instance 
78           # of foo until you don't need it anymore! 
79           foo = Foo() 
80           foo.start() 
81

83   .. _schedule-before-frame: 
84    
85   Schedule before frame 
86   --------------------- 
87    
88   .. versionadded:: 1.0.5 
89    
90   Sometimes you need to schedule a callback BEFORE the next frame. Starting 
91   from 1.0.5, you can use a timeout of -1:: 
92    
93       Clock.schedule_once(my_callback, 0) # call after the next frame 
94       Clock.schedule_once(my_callback, -1) # call before the next frame 
95    
96   The Clock will execute all the callbacks with a timeout of -1 before the 
97   next frame even if you add a new callback with -1 from a running 
98   callback. However, :class:`Clock` has an iteration limit for these 
99   callbacks: it defaults to 10. 
100   
101  If you schedule a callback that schedules a callback that schedules a ... etc 
102  more than 10 times, it will leave the loop and send a warning to the console, 
103  then continue after the next frame. This is implemented to prevent bugs from 
104  hanging or crashing the application. 
105   
106  If you need to increase the limit, set the :attr:`max_iteration` property:: 
107   
108      from kivy.clock import Clock 
109      Clock.max_iteration = 20 
110   
111  .. _triggered-events: 
112   
113  Triggered Events 
114  ---------------- 
115   
116  .. versionadded:: 1.0.5 
117   
118  :meth:`CyClockBase.create_trigger` is an advanced method way to defer a 
119  callback. It functions exactly like :meth:`CyClockBase.schedule_once` and 
120  :meth:`CyClockBase.schedule_interval` except that it doesn't immediately 
121  schedule the callback. Instead, one schedules the callback using the 
122  :class:`ClockEvent` returned by it. This ensures that you can call the event 
123  multiple times but it won't be scheduled more than once. This is not the case 
124  with :meth:`CyClockBase.schedule_once`:: 
125   
126      # will run the callback twice before the next frame 
127      Clock.schedule_once(my_callback) 
128      Clock.schedule_once(my_callback) 
129   
130      # will run the callback once before the next frame 
131      event = Clock.create_trigger(my_callback) 
132      event() 
133      event() 
134   
135      # will also run the callback only once before the next frame 
136      event = Clock.schedule_once(my_callback)  # now it's already scheduled 
137      event()  # won't be scheduled again 
138      event() 
139   
140  In addition, it is more convenient to create and bind to 
141  the triggered event than using :meth:`CyClockBase.schedule_once` in a 
142  function:: 
143   
144      from kivy.clock import Clock 
145      from kivy.uix.widget import Widget 
146   
147      class Sample(Widget): 
148          def __init__(self, **kwargs): 
149              self._trigger = Clock.create_trigger(self.cb) 
150              super(Sample, self).__init__(**kwargs) 
151              self.bind(x=self._trigger, y=self._trigger) 
152   
153          def cb(self, *largs): 
154              pass 
155   
156  Even if x and y changes within one frame, the callback is only run once. 
157   
158  Unscheduling 
159  ------------- 
160   
161  An event scheduled with :meth:`CyClockBase.schedule_once`, 
162  :meth:`CyClockBase.schedule_interval`, or with 
163  :meth:`CyClockBase.create_trigger` and then triggered can be unscheduled in 
164  multiple ways. E.g:: 
165   
166      def my_callback(dt): 
167          pass 
168   
169      # call my_callback every 0.5 seconds 
170      event = Clock.schedule_interval(my_callback, 0.5) 
171   
172      # call my_callback in 5 seconds 
173      event2 = Clock.schedule_once(my_callback, 5) 
174   
175      event_trig = Clock.create_trigger(my_callback, 5) 
176      event_trig() 
177   
178      # unschedule using cancel 
179      event.cancel() 
180   
181      # unschedule using Clock.unschedule 
182      Clock.unschedule(event2) 
183   
184      # unschedule using Clock.unschedule with the callback 
185      # NOT RECOMMENDED 
186      Clock.unschedule(my_callback) 
187   
188  The best way to unschedule a callback is with :meth:`ClockEvent.cancel`. 
189  :meth:`CyClockBase.unschedule` is mainly an alias for that for that function. 
190  However, if the original callback itself is passed to 
191  :meth:`CyClockBase.unschedule`, it'll unschedule all instances of that 
192  callback (provided ``all`` is True, the default, otherwise only the first match 
193  is removed). 
194   
195  Calling :meth:`CyClockBase.unschedule` on the original callback is highly 
196  discouraged because it's significantly slower than when using the event. 
197   
198  Clock Lifecycle 
199  --------------- 
200   
201  Kivy's clock has a lifecycle. By default, scheduling a callback after the Clock 
202  has ended will not raise an error, even though the callback may never be 
203  called. That's because most callbacks are like services, e.g. responding to a 
204  user button press - if the app is running the callbacks need to service the app 
205  and respond to the input, but once the app has stopped or is stopping, we can 
206  safely not process these events. 
207   
208  Other events always need to be processed. E.g. another thread may request a 
209  callback in kivy's thread and then process some result. If the event is not 
210  processed in Kivy's thread because the app stopped, the second thread may 
211  block forever hanging the application as it exits. 
212   
213  Consequently, we provide a API 
214  (:meth:`CyClockBase.create_lifecycle_aware_trigger`) for scheduling callbacks 
215  that raise a :class:`ClockNotRunningError` if the clock has stopped. If the 
216  scheduling succeeded it guarantees that one of its callbacks will be called. 
217  I.e. the new :meth:`CyClockBase.create_lifecycle_aware_trigger` accepts an 
218  additional ``clock_ended_callback`` parameter. Normally, ``callback`` will be 
219  called when the event is processed. But, if the clock is stopped before it can 
220  be processed, if the application exited normally (and the app was started) and 
221  the event wasn't canceled, and the callbacks are not garbage collected, then 
222  ``clock_ended_callback`` will be called instead when the clock is stopped. 
223   
224  That is, given these conditions, if :class:`ClockNotRunningError` was not 
225  raised when the event was scheduled, then one of these callbacks will be 
226  called - either ``callback`` if the event executed normally, or 
227  ``clock_ended_callback`` if the clock is stopped while the event is scheduled. 
228   
229  By default, events can be scheduled before the clock is started because it is 
230  assumed the clock will eventually be started when the app starts. I.e. 
231  calling :meth:`CyClockBase.create_lifecycle_aware_trigger` before the clock 
232  and application starts will succeed. But if the app never actually starts, then 
233  neither of the callbacks may be executed. 
234   
235  .. versionadded:: 2.0.0 
236      The lifecycle was added in 2.0.0 
237   
238  Exception Handling 
239  ------------------ 
240   
241  Kivy provides a exception handling manager, 
242  :attr:`~kivy.base.ExceptionManager`, to handle its internal exceptions 
243  including exceptions raised by clock callbacks, without crashing the 
244  application. By default when an exception is raised, the app will crash. 
245  But, if a handler is registered with the exception manager and the handler 
246  handles the exception, the app will not crash and will continue as normal.:: 
247   
248      from kivy.base import ExceptionHandler, ExceptionManager 
249      class MyHandler(ExceptionHandler): 
250          def handle_exception(self, inst): 
251              if isinstance(inst, ValueError): 
252                  Logger.exception('ValueError caught by MyHandler') 
253                  return ExceptionManager.PASS 
254              return ExceptionManager.RAISE 
255   
256      ExceptionManager.add_handler(MyHandler()) 
257   
258  Then, all ValueError exceptions will be logged to the console and ignored. 
259  Similarly, if a scheduled clock callback raises a ValueError, other clock 
260  events will still be processed normally. 
261   
262  If an event's callback raises an exception, before the exception handler is 
263  executed, the callback is immediately canceled. 
264   
265  It still is possible for the app to be corrupted if kivy itself is the source 
266  of the exception. I.e. even with a handler that ignores exceptions and doesn't 
267  crash, the app may be in a corrupted state if the error originates from within 
268  Kivy itself. However, the exception handler can help protect the app from 
269  crashing and can help protect against user callbacks crashing the app. 
270   
271  .. versionchanged:: 2.0.0 
272      Prior to Kivy 2.0.0, an exception raised in a event's callback would 
273      cause the clock to crash and subsequent events may or may not be executed. 
274      Even if the exception was handled by an 
275      :class:`~kivy.base.ExceptionHandler`, there was no guarantee that some 
276      scheduled events would not be skipped. 
277   
278      From 2.0.0 onward, if a event's exception is handled by an 
279      :class:`~kivy.base.ExceptionHandler`, other events will be shielded from 
280      the exception and will execute normally. 
281   
282  Scheduling from ``__del__`` 
283  --------------------------- 
284   
285  It is not safe to schedule Clock events from a object's ``__del__`` or 
286  ``__dealloc__`` method. If you must schedule a Clock call from this method, use 
287  :meth:`CyClockBase.schedule_del_safe` or 
288  :meth:`CyClockBase.schedule_lifecycle_aware_del_safe` instead. 
289   
290  Threading and Callback Order 
291  ----------------------------- 
292   
293  Beginning with 1.10.0, all the events scheduled for the same frame, e.g. 
294  all the events scheduled in the same frame with a ``timeout`` of ``0``, 
295  well be executed in the order they were scheduled. 
296   
297  Also, all the scheduling and canceling methods are fully thread safe and 
298  can be safely used from external threads. 
299   
300  As a a consequence, calling :meth:`CyClockBase.unschedule` with the original 
301  callback is now significantly slower and highly discouraged. Instead, the 
302  returned events should be used to cancel. As a tradeoff, all the other methods 
303  are now significantly faster than before. 
304   
305  Advanced Clock Details 
306  ----------------------- 
307   
308  The following section goes into the internal kivy clock details as well 
309  as the various clock options. It is meant only for advanced users. 
310   
311  Fundamentally, the Kivy clock attempts to execute any scheduled callback 
312  rhythmically as determined by the specified fps (frame per second, see 
313  ``maxfps`` in :mod:`~kivy.config`). That is, ideally, given e.g. a desired fps 
314  of 30, the clock will execute the callbacks at intervals of 1 / 30 seconds, or 
315  every 33.33 ms. All the callbacks in a frame are given the same timestamp, 
316  i.e. the ``dt`` passed to the callback are all the same and it's the difference 
317  in time between the start of this and the previous frame. 
318   
319  Because of inherent indeterminism, the frames do not actually occur exactly 
320  at intervals of the fps and ``dt`` may be under or over the desired fps. 
321  Also, once the timeout is "close enough" to the desired timeout, as determined 
322  internally, Kivy will execute the callback in the current frame even when the 
323  "actual time" has not elapsed the ``timeout`` amount. 
324   
325  Kivy offers now, since ``1.10.0``, multiple clocks with different behaviors. 
326   
327  Default Clock 
328  ^^^^^^^^^^^^^^ 
329   
330  The default clock (``default``) behaves as described above. When a callback 
331  with a timeout of zero or non-zero is scheduled, they are executed at the frame 
332  that is near the timeout, which is a function of the fps. So a timeout of zero 
333  would still result in a delay of one frame or about 1 / fps, typically a bit 
334  less but sometimes more depending on the CPU usage of the other events 
335  scheduled for that frame. 
336   
337  In a test using a fps of 30, a callback with a timeout of 0, 0.001, and 0.05, 
338  resulted in a mean callback delay of 0.02487, 0.02488, and 0.05011 seconds, 
339  respectively. When tested with a fps of 600 the delay for 0.05 was similar, 
340  except the standard deviation was reduced resulting in overall better accuracy. 
341   
342  Interruptible Clock 
343  ^^^^^^^^^^^^^^^^^^^^ 
344   
345  The default clock suffers from the quantization problem, as frames occur only 
346  on intervals and any scheduled timeouts will not be able to occur during an 
347  interval. For example, with the timeout of 0.05, while the mean was 0.05011, 
348  its values ranged between 0.02548 - 0.07348 and a standard deviation of 0.002. 
349  Also, there's the minimum timeout of about 0.02487. 
350   
351  The interruptible clock (``interrupt``) will execute timeouts even during a 
352  frame. So a timeout of zero will execute as quickly as possible and similarly 
353  a non-zero timeout will be executed even during the interval. 
354   
355  This clock, and all the clocks described after this have an option, 
356  :attr:`ClockBaseInterruptBehavior.interupt_next_only`. When True, any of the 
357  behavior new behavior will only apply to the callbacks with a timeout of 
358  zero. Non-zero timeouts will behave like in the default clock. E.g. for this 
359  clock when True, only zero timeouts will execute during the the interval. 
360   
361  In a test using a fps of 30, a callback with a timeout of 0, 0.001, and 0.05, 
362  resulted in a mean callback delay of 0.00013, 0.00013, and 0.04120 seconds, 
363  respectively when :attr:`ClockBaseInterruptBehavior.interupt_next_only` was 
364  False. Also, compared to the default clock the standard deviation was reduced. 
365  When :attr:`ClockBaseInterruptBehavior.interupt_next_only` was True, the values 
366  were 0.00010, 0.02414, and 0.05034, respectively. 
367   
368  Free Clock 
369  ^^^^^^^^^^^ 
370   
371  The interruptible clock may not be ideal for all cases because all the events 
372  are executed during the intervals and events are not executed anymore 
373  rhythmically as multiples of the fps. For example, there may not be any benefit 
374  for the graphics to update in a sub-interval, so the additional accuracy 
375  wastes CPU. 
376   
377  The Free clock (``free_all``) solves this by having ``Clock.xxx_free`` versions 
378  of all the Clock scheduling methods. By free, we mean the event is free from 
379  the fps because it's not fps limited. E.g. 
380  :meth:`CyClockBaseFree.create_trigger_free` corresponds to 
381  :meth:`CyClockBase.create_trigger`. Only when an event scheduled using the 
382  ``Clock.xxx_free`` methods is present will the clock interrupt and execute 
383  the events during the interval. So, if no ``free`` event is present the clock 
384  behaves like the ``default`` clock, otherwise it behaves like the ``interrupt`` 
385  clock. 
386   
387  In a test using a fps of 30, a callback with a timeout of 0s, 0.001s, and 
388  0.05s, resulted in a mean callback delay of 0.00012s, 0.00017s, and 0.04121s 
389  seconds, respectively when it was a free event and 0.02403s, 0.02405s, and 
390  0.04829s, respectively when it wasn't. 
391   
392  Free Only Clock 
393  ^^^^^^^^^^^^^^^^^ 
394   
395  The Free clock executes all events when a free event was scheduled. This 
396  results in normal events also being execute in the middle of the interval 
397  when a free event is scheduled. For example, above, when a free event was 
398  absent, a normal event with a 0.001s timeout was delayed for 0.02405s. However, 
399  if a free event happened to be also scheduled, the normal event was only 
400  delayed 0.00014s, which may be undesirable. 
401   
402  The Free only clock (``free_only``) solves it by only executing free events 
403  during the interval and normal events are always executed like with the 
404  default clock. For example, in the presence of a free event, a normal event 
405  with a timeout of 0.001s still had a delay of 0.02406. So this clock, 
406  treats free and normal events independently, with normal events always being 
407  fps limited, but never the free events. 
408   
409  Summary 
410  ^^^^^^^^ 
411   
412  The kivy clock type to use can be set with the ``kivy_clock`` option the 
413  :mod:`~kivy.config`. If ``KIVY_CLOCK`` is present in the environment it 
414  overwrites the config selection. Its possible values are as follows: 
415   
416  * When ``kivy_clock`` is ``default``, the normal clock, :class:`ClockBase`, 
417    which limits callbacks to the maxfps quantization - is used. 
418  * When ``kivy_clock`` is ``interrupt``, a interruptible clock, 
419    :class:`ClockBaseInterrupt`, which doesn't limit any callbacks to the 
420    maxfps - is used. Callbacks will be executed at any time. 
421  * When ``kivy_clock`` is ``free_all``, a interruptible clock, 
422    :class:`ClockBaseFreeInterruptAll`, which doesn't limit any callbacks to the 
423    maxfps in the presence of free events, but in their absence it limits events 
424    to the fps quantization interval - is used. 
425  * When ``kivy_clock`` is ``free_only``, a interruptible clock, 
426    :class:`ClockBaseFreeInterruptAll`, which treats free and normal events 
427    independently; normal events are fps limited while free events are not - is 
428    used. 
429   
430  Async clock support 
431  ------------------- 
432   
433  .. versionadded:: 2.0.0 
434   
435  Experimental async support has been added in 2.0.0. The Clock now has a 
436  :meth:`ClockBaseBehavior.async_tick` and :meth:`ClockBaseBehavior.async_idle` 
437  coroutine method which is used by the kivy EventLoop when the kivy EventLoop is 
438  executed in a asynchronous manner. When used, the kivy clock does not 
439  block while idling. 
440   
441  The async library to use is selected with the `KIVY_EVENTLOOP` environmental 
442  variable or by  calling :meth:`~kivy.clock.ClockBaseBehavior.init_async_lib` 
443  directly. The library can be one of `"asyncio"` when the standard library 
444  `asyncio` should be used, or `"trio"` if the trio library 
445  should be used. If not set it defaults to `"asyncio"`. 
446   
447  See :mod:`~kivy.app` for example usage.

607      SLEEP_UNDERSHOOT = MIN_SLEEP - 0.001
608  
609      _async_lib = None
610  
611      _async_wait_for = None
612  
613      def __init__(self, async_lib='asyncio', **kwargs):
614          self.init_async_lib(async_lib)
615          super(ClockBaseBehavior, self).__init__(**kwargs)
616          self._duration_ts0 = self._start_tick = self._last_tick = self.time()
617          self._max_fps = float(Config.getint('graphics', 'maxfps'))

619      def init_async_lib(self, lib):
620          """Manually sets the async library to use internally, when running in 
621          a asynchronous manner. 
622   
623          This can be called anytime before the kivy event loop has started, 
624          but not once the kivy App is running. 
625   
626          :parameters: 
627   
628              `lib`: string 
629                  The async library to use when the clock is run asynchronously. 
630                  Can be one of, `"asyncio"` when the standard library asyncio 
631                  should be used, or `"trio"` if the trio library should be used. 
632          """
633          if lib == 'trio':
634              import trio
635              self._async_lib = trio
636  
637              async def wait_for(coro, t):
638                  with trio.move_on_after(t):
639                      await coro
640              self._async_wait_for = wait_for
641          elif lib == 'asyncio':
642              import asyncio
643              self._async_lib = asyncio
644              self._async_wait_for = asyncio.wait_for
645          else:
646              raise ValueError('async library {} not recognized'.format(lib))

677      def idle(self):
678          '''(internal) waits here until the next frame. 
679          '''
680          fps = self._max_fps
681          if fps > 0:
682              min_sleep = self.get_resolution()
683              undershoot = 4 / 5. * min_sleep
684              usleep = self.usleep
685              ready = self._check_ready
686  
687              done, sleeptime = ready(fps, min_sleep, undershoot)
688              while not done:
689                  usleep(1000000 * sleeptime)
690                  done, sleeptime = ready(fps, min_sleep, undershoot)
691  
692          current = self.time()
693          self._dt = current - self._last_tick
694          self._last_tick = current
695          return current
696  
697      async def async_idle(self):
698          '''(internal) async version of :meth:`idle`. 
699          '''
700          fps = self._max_fps
701          if fps > 0:
702              min_sleep = self.get_resolution()
703              undershoot = 4 / 5. * min_sleep
704              ready = self._check_ready
705  
706              slept = False
707              done, sleeptime = ready(fps, min_sleep, undershoot)
708              while not done:
709                  slept = True
710                  await self._async_lib.sleep(sleeptime)
711                  done, sleeptime = ready(fps, min_sleep, undershoot)
712  
713              if not slept:
714                  await self._async_lib.sleep(0)
715          else:
716              await self._async_lib.sleep(0)
717  
718          current = self.time()
719          self._dt = current - self._last_tick
720          self._last_tick = current
721          return current
722  
723      def _check_ready(self, fps, min_sleep, undershoot):
724          sleeptime = 1 / fps - (self.time() - self._last_tick)
725          return sleeptime - undershoot <= min_sleep, sleeptime - undershoot
726  
727      def tick(self):
728          '''Advance the clock to the next step. Must be called every frame. 
729          The default clock has a tick() function called by the core Kivy 
730          framework.'''
731          self.pre_idle()
732          ts = self.time()
733          self.post_idle(ts, self.idle())
734  
735      async def async_tick(self):
736          '''async version of :meth:`tick`. '''
737          self.pre_idle()
738          ts = self.time()
739          current = await self.async_idle()
740          self.post_idle(ts, current)

742      def pre_idle(self):
743          '''Called before :meth:`idle` by :meth:`tick`. 
744          '''
745          self._release_references()
746  
747      def post_idle(self, ts, current):
748          '''Called after :meth:`idle` by :meth:`tick`. 
749          '''
750          # tick the current time
751          self._frames += 1
752          self._fps_counter += 1
753  
754          # compute how long the event processing takes
755          self._duration_count += 1
756          self._sleep_time += current - ts
757          t_tot = current - self._duration_ts0
758          if t_tot >= 1.:
759              self._events_duration = \
760                  (t_tot - self._sleep_time) / float(self._duration_count)
761              self._duration_ts0 = current
762              self._sleep_time = self._duration_count = 0
763  
764          # calculate fps things
765          if self._last_fps_tick is None:
766              self._last_fps_tick = current
767          elif current - self._last_fps_tick > 1:
768              d = float(current - self._last_fps_tick)
769              self._fps = self._fps_counter / d
770              self._rfps = self._rfps_counter
771              self._last_fps_tick = current
772              self._fps_counter = 0
773              self._rfps_counter = 0
774  
775          # process event
776          self._process_events()
777  
778          return self._dt
779  
780      def tick_draw(self):
781          '''Tick the drawing counter. 
782          '''
783          self._process_events_before_frame()
784          self._rfps_counter += 1
785          self._frames_displayed += 1
786  
787      def get_fps(self):
788          '''Get the current average FPS calculated by the clock. 
789          '''
790          return self._fps
791  
792      def get_rfps(self):
793          '''Get the current "real" FPS calculated by the clock. 
794          This counter reflects the real framerate displayed on the screen. 
795   
796          In contrast to get_fps(), this function returns a counter of the 
797          number of frames, not the average of frames per second. 
798          '''
799          return self._rfps

801      def get_time(self):
802          '''Get the last tick made by the clock.'''
803          return self._last_tick
804  
805      def get_boottime(self):
806          '''Get the time in seconds from the application start.'''
807          return self._last_tick - self._start_tick

822  class ClockBaseInterruptBehavior(ClockBaseBehavior):
823      '''A kivy clock which can be interrupted during a frame to execute events. 
824      '''
825  
826      interupt_next_only = False
827      _event = None
828      _async_event = None
829      _get_min_timeout_func = None
830  
831      def __init__(self, interupt_next_only=False, **kwargs):
832          super(ClockBaseInterruptBehavior, self).__init__(**kwargs)
833          self._event = ThreadingEvent()
834          self.interupt_next_only = interupt_next_only
835          self._get_min_timeout_func = self.get_min_timeout
836  
837      def init_async_lib(self, lib):
838          super(ClockBaseInterruptBehavior, self).init_async_lib(lib)
839          if lib == 'trio':
840              import trio
841              self._async_event = trio.Event()
842              # we don't know if this is called after things have already been
843              # scheduled, so don't delay for a full frame before processing
844              # events
845              self._async_event.set()
846          elif lib == 'asyncio':
847              import asyncio
848              self._async_event = asyncio.Event()
849              self._async_event.set()
850  
851      def usleep(self, microseconds):
852          self._event.clear()
853          self._event.wait(microseconds / 1000000.)
854  
855      async def async_usleep(self, microseconds):
856          self._async_event.clear()
857          await self._async_wait_for(
858              self._async_event.wait(), microseconds / 1000000.)
859  
860      def on_schedule(self, event):
861          fps = self._max_fps
862          if not fps:
863              return
864  
865          if not event.timeout or (
866                  not self.interupt_next_only and event.timeout <=
867                  1 / fps -  # remaining time
868                  (self.time() - self._last_tick) +  # elapsed time
869                  4 / 5. * self.get_resolution()):  # resolution fudge factor
870              self._event.set()
871              if self._async_event:
872                  self._async_event.set()
873  
874      def idle(self):
875          fps = self._max_fps
876          event = self._event
877          resolution = self.get_resolution()
878          if fps > 0:
879              done, sleeptime = self._check_ready(
880                  fps, resolution, 4 / 5. * resolution, event)
881              if not done:
882                  event.wait(sleeptime)
883  
884          current = self.time()
885          self._dt = current - self._last_tick
886          self._last_tick = current
887          event.clear()
888          # anything scheduled from now on, if scheduled for the upcoming frame
889          # will cause a timeout of the event on the next idle due to on_schedule
890          # `self._last_tick = current` must happen before clear, otherwise the
891          # on_schedule computation is wrong when exec between the clear and
892          # the `self._last_tick = current` bytecode.
893          return current
894  
895      async def async_idle(self):
896          fps = self._max_fps
897          event = self._async_event
898          resolution = self.get_resolution()
899          if fps > 0:
900              done, sleeptime = self._check_ready(
901                  fps, resolution, 4 / 5. * resolution, event)
902              if not done:
903                  await self._async_wait_for(event.wait(), sleeptime)
904              else:
905                  await self._async_lib.sleep(0)
906          else:
907              await self._async_lib.sleep(0)
908  
909          current = self.time()
910          self._dt = current - self._last_tick
911          self._last_tick = current
912          event.clear()
913          # anything scheduled from now on, if scheduled for the upcoming frame
914          # will cause a timeout of the event on the next idle due to on_schedule
915          # `self._last_tick = current` must happen before clear, otherwise the
916          # on_schedule computation is wrong when exec between the clear and
917          # the `self._last_tick = current` bytecode.
918          return current
919  
920      def _check_ready(self, fps, min_sleep, undershoot, event):
921          if event.is_set():
922              return True, 0
923  
924          t = self._get_min_timeout_func()
925          if not t:
926              return True, 0
927  
928          if not self.interupt_next_only:
929              curr_t = self.time()
930              sleeptime = min(1 / fps - (curr_t - self._last_tick), t - curr_t)
931          else:
932              sleeptime = 1 / fps - (self.time() - self._last_tick)
933          return sleeptime - undershoot <= min_sleep, sleeptime - undershoot

1073 def mainthread(func):
1074     '''Decorator that will schedule the call of the function for the next 
1075     available frame in the mainthread. It can be useful when you use 
1076     :class:`~kivy.network.urlrequest.UrlRequest` or when you do Thread 
1077     programming: you cannot do any OpenGL-related work in a thread. 
1078  
1079     Please note that this method will return directly and no result can be 
1080     returned:: 
1081  
1082         @mainthread 
1083         def callback(self, *args): 
1084             print('The request succeeded!', 
1085                   'This callback is called in the main thread.') 
1086  
1087  
1088         self.req = UrlRequest(url='http://...', on_success=callback) 
1089  
1090     .. versionadded:: 1.8.0 
1091     '''
1092     @wraps(func)
1093     def delayed_func(*args, **kwargs):
1094         def callback_func(dt):
1095             func(*args, **kwargs)
1096         Clock.schedule_once(callback_func, 0)
1097     return delayed_func

1100 def triggered(timeout=0, interval=False):
1101     '''Decorator that will trigger the call of the function at the specified 
1102     timeout, through the method :meth:`CyClockBase.create_trigger`. Subsequent 
1103     calls to the decorated function (while the timeout is active) are ignored. 
1104  
1105     It can be helpful when an expensive funcion (i.e. call to a server) can be 
1106     triggered by different methods. Setting a proper timeout will delay the 
1107     calling and only one of them wil be triggered. 
1108  
1109         @triggered(timeout, interval=False) 
1110         def callback(id): 
1111             print('The callback has been called with id=%d' % id) 
1112  
1113         >> callback(id=1) 
1114         >> callback(id=2) 
1115         The callback has been called with id=2 
1116  
1117     The decorated callback can also be unscheduled using: 
1118  
1119         >> callback.cancel() 
1120  
1121     .. versionadded:: 1.10.1 
1122     '''
1123 
1124     def wrapper_triggered(func):
1125 
1126         _args = []
1127         _kwargs = {}
1128 
1129         def cb_function(dt):
1130             func(*tuple(_args), **_kwargs)
1131 
1132         cb_trigger = Clock.create_trigger(
1133             cb_function,
1134             timeout=timeout,
1135             interval=interval)
1136 
1137         @wraps(func)
1138         def trigger_function(*args, **kwargs):
1139             _args[:] = []
1140             _args.extend(list(args))
1141             _kwargs.clear()
1142             _kwargs.update(kwargs)
1143             cb_trigger()
1144 
1145         def trigger_cancel():
1146             cb_trigger.cancel()
1147 
1148         setattr(trigger_function, 'cancel', trigger_cancel)
1149 
1150         return trigger_function
1151 
1152     return wrapper_triggered
